# The Last Mile — Technical Design

## 1) Purpose and Product Scope

The Last Mile is an AI-assisted estate administration application for executors and surviving family members. It transforms intake answers into a deterministic task plan, then layers AI assistance for drafting, enrichment, and contextual guidance with strict legal and privacy guardrails.

The design goal is to provide practical automation without surrendering control: the system can draft and guide, but cannot take irreversible actions (for example, it cannot send communications on a user's behalf).

## 2) System Overview

At a high level, the product is a browser-first app with a thin Node/Express backend proxy.

- Frontend (`passage-v2.html` + `js/*`) handles UX, state, local persistence, workflows, and most orchestration.
- Backend (`server/index.js`) serves static assets, proxies Anthropic API requests, and provides a specialized streaming bank-draft orchestration endpoint.
- AI model usage is centralized around Anthropic Sonnet for generation and enrichment tasks.
- Privacy and scope controls are layered across deterministic checks, prompt constraints, regex verification, and manual user confirmation.

## 3) Core Design Principles

- Deterministic core, AI-assisted detail: the checklist is generated by rules from user intake, never hallucinated by an LLM.
- Privacy first: sensitive user context remains in-browser by default with explicit obfuscation before document parsing prompts.
- Human-in-the-loop execution: all externally-facing drafts require user review and manual sending.
- Guardrails before capability: off-scope or high-risk requests are blocked before model invocation.
- Full traceability: each major agent action is logged with human-readable status, legal basis references, and source URLs.

## 4) Architecture Components

### Frontend Modules

- `js/main.js`: app bootstrap and API connectivity behavior.
- `js/state.js`: global application state, `localStorage` persistence, and request helpers (`anthropicFetch()`).
- `js/tasks.js`: deterministic task generation, AI enrichment, and complexity checks.
- `js/agents.js`: trace event infrastructure and send-review gate wiring.
- `js/guardrails.js`: pattern-based scope and sensitivity checks.
- `js/onboarding.js`: intake flow, file handling, and PII obfuscation path.
- `js/chat.js`: context-aware assistant chat, streaming response render, and source extraction.
- `js/render.js` + `js/workflows/*`: task workflow UIs, letter/call script generation paths.
- `js/knowledge.js`: curated legal bases and official source mappings.

### Backend Modules

- `server/index.js`: static hosting, health endpoint, model proxy endpoint, and bank orchestration endpoint.
- `server/knowledge/banks.json`: curated institution contact knowledge.
- `server/lib/knowledge.js`: lookup helper for institution data.
- `server/lib/guardrails.js`: server-side blocked phrase checks for additional defense-in-depth.

## 5) End-to-End Workflow

1. User completes intake and optional uploads.
2. Task Generation Agent creates a personalized checklist using deterministic rules.
3. Each task is assigned curated legal basis and official sources.
4. AI Enrichment Agent enhances eligible tasks with state-specific operational details.
5. Complexity Check Agent evaluates high-risk conditions and records warnings.
6. User selects a task and initiates a workflow (letter generation, call script, etc.).
7. General Assistant guardrail check validates scope before any LLM call.
8. Communication/Chat agent streams model output to the UI.
9. Verify Agent scans generated draft for sensitive data patterns.
10. User must explicitly review and confirm before manual external sending.

## 6) Agentic Design

The system is intentionally split into bounded agents to reduce blast radius:

- Deterministic agents own checklist correctness and risk flagging.
- LLM agents are constrained to enrichment, drafting, extraction, and conversational support.
- Verification and send gating are non-LLM and enforceable even if model output is imperfect.

This produces a hybrid architecture where predictable legal workflow logic and probabilistic language generation are separated by policy boundaries.

## 7) Data Model and State Strategy

### Client State

Primary state lives in browser memory with resume support via `localStorage`:

- Intake profile (relationship, state, asset and survivor context).
- Task list, completion status, and workflow outputs.
- Chat history and contextual references.
- Temporary parsing artifacts for upload extraction.

### Persistence Characteristics

- No required database for MVP operation.
- Session continuity is handled client-side.
- This keeps deployment simple while supporting rapid iteration and low operational overhead.

## 8) AI Integration Pattern

Anthropic Sonnet is used for:

- Task enrichment (state-specific details + source URL requirement).
- Letter and call script drafting.
- HR contact lookup assistance.
- Attorney/CPA recommendation drafting.
- Contextual chat responses with required citation line.
- Document parsing of obfuscated text.

Streaming is first-class for user trust and responsiveness: users can see generation as it happens and interrupt/inspect in near real time.

## 9) Guardrail and Safety Architecture

### Scope Guardrails

- Pattern checks block requests outside product policy (for example: asset hiding, tax evasion, will contests).
- Guardrail checks run before LLM calls in user-facing AI paths.

### PII and Sensitive Data Controls

- Prompt-level prohibition (`COMM_PII_RULE`) for communication generation.
- Upload text obfuscation before parsing requests.
- Pre-send regex verification for SSNs, account numbers, DOBs, and credential-like strings.
- Severity-based user acknowledgment model in the send flow.

### Human Oversight

- No automatic outbound action by design.
- Users must manually approve and manually send communications.

### Explainability and Auditability

- Agent Trace records key steps, warnings, and references.
- Legal and source citation hooks make rationale inspectable for users and reviewers.

## 10) API Surface

### `GET /health`

Simple liveness check for local/dev and hosted readiness.

### `POST /v1/messages`

Backend proxy to Anthropic API. Keeps keys server-side in production and enables streaming responses.

### `POST /api/orchestrate/bank-draft`

Specialized orchestration endpoint that streams Server-Sent Events (`trace`, `draft_delta`, `done`) for bank workflow generation.

## 11) Reliability and Failure Handling

- Deterministic fallback: if enrichment fails, task descriptions remain usable.
- Guardrail-first behavior: blocked requests fail safely and explain why.
- Streaming UX resilience: failure in generation yields explicit error messaging without corrupting prior state.
- Thin backend: reduced stateful failure modes and easier incident recovery.

## 12) Security and Privacy Posture (Current State)

- API key isolation in backend deployment mode (`ANTHROPIC_API_KEY`).
- Minimal server data retention by architecture (client-first state).
- No automatic dispatch of potentially sensitive communications.
- Pattern-based sensitivity checks before final user action.

For future hardening, enterprise deployment can add encryption-at-rest persistence, scoped session tokens, and centralized audit logging while preserving the same control boundaries.

## 13) Deployment Topology

### Local

- Single command startup (`npm start`) installs backend deps and serves app on port `8787`.
- Optional browser-key mode supported for purely local prototyping.

### Production

- Frontend can be served statically.
- Backend can run on Node hosts (for proxy + orchestration).
- `PASSAGE_API_BASE` enables frontend/backend separation when needed.

This separation allows low-cost static hosting while preserving secure server-side key management.

## 14) Scalability Roadmap

The architecture is positioned for iterative scale:

- Introduce managed persistence for collaborative or multi-device execution.
- Add tenant/org boundaries and RBAC for professional workflows.
- Move selected rule engines into versioned policy services.
- Expand event tracing into analytics and compliance dashboards.
- Add retrieval-backed legal content updates while keeping deterministic task generation logic isolated.

## 15) Why This Design Fits Investor Priorities

- Fast time-to-value: lightweight stack, low infra complexity, clear onboarding-to-output funnel.
- Defensible trust layer: deterministic legal workflow backbone plus multi-layer guardrails.
- Operational efficiency: browser-first architecture minimizes backend load and storage complexity.
- Expansion-ready: clean module boundaries around workflows and agents support new verticals (financial institutions, legal partnerships, survivor benefit automation).

## 16) Demo Narrative (Suggested)

For an investor walkthrough, this sequence best highlights technical leverage:

1. Intake -> deterministic task generation (show non-LLM reliability).
2. AI enrichment on tasks (show state-specific value add).
3. Letter drafting stream (show user-visible AI productivity gain).
4. Verify/send gate and trace log (show trust, safety, and auditability).
5. Chat with sources (show guided support without over-automation).

This narrative makes the core thesis clear: a safety-first agentic architecture can deliver meaningful automation in a high-trust legal workflow domain.
